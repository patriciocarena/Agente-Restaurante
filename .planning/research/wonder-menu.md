## Hallazgos clave para el schema

Items sin `price` base (con `_(precio variable)_` arriba) tienen el precio determinado por la opción elegida. Nuestro `menu_items.price numeric` actual asume un precio único; hay que soportar precios condicionales o convertir cada variante en un item separado.

Los grupos de opciones tienen cardinalidad (min/max selecciones) — nuestro `modifiers jsonb` plano no lo expresa. Hace falta estructura tipo:

```json
{
  "options": [
    {
      "name": "Bebida",
      "min": 1,
      "max": 1,
      "choices": [{"name": "Pepsi", "price_delta": 0}, ...]
    },
    {
      "name": "Toppings",
      "min": 0,
      "max": 8,
      "choices": [{"name": "Bacon", "price_delta": 0}, ...]
    }
  ]
}
```

Las promociones complejas (ej. 'Martes de 2x1 LITERAL') requieren modelar **dos hamburguesas con sus toppings independientes**. Para v1 sugiero excluir promociones del onboarding inicial, manejarlas como items pre-armados con un solo grupo de opciones.
