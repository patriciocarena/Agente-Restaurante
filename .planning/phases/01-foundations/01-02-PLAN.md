```sql
-- supabase/tests/rls-tenant-isolation.sql
-- Verifies AUTH-05 cross-tenant isolation. Run via Supabase Dashboard SQL Editor
-- as a privileged role; the script switches into the authenticated role and
-- impersonates two different tenants to prove RLS blocks cross-reads.
--
-- Expected output: every assertion query returns 0 rows.

-- Setup: two restaurants owned by two distinct fake user UUIDs
DO $$
DECLARE
  user_a uuid := '00000000-0000-0000-0000-000000000aaa';
  user_b uuid := '00000000-0000-0000-0000-000000000bbb';
  rest_a uuid;
  rest_b uuid;
BEGIN
  -- Insert two restaurants directly (bypassing auth.users — for isolation test only)
  INSERT INTO restaurants (owner_id, name) VALUES (user_a, 'Tenant A Test') RETURNING id INTO rest_a;
  INSERT INTO restaurants (owner_id, name) VALUES (user_b, 'Tenant B Test') RETURNING id INTO rest_b;
  -- Insert one menu_item for each
  INSERT INTO menu_items (restaurant_id, name, base_price) VALUES (rest_a, 'Burger A', 5000);
  INSERT INTO menu_items (restaurant_id, name, base_price) VALUES (rest_b, 'Burger B', 6000);
  RAISE NOTICE 'Setup complete: rest_a=%, rest_b=%', rest_a, rest_b;
END $$;

-- Switch to authenticated role with tenant A's claim
SET ROLE authenticated;
SET request.jwt.claims = '{"app_metadata":{"restaurant_id":"<REST_A_UUID>"}}';
-- ASSERT: tenant A sees only its own row in restaurants
SELECT count(*) AS expected_1, (SELECT count(*) FROM restaurants) AS actual;
-- ASSERT: tenant A sees only its own menu_items
SELECT count(*) AS expected_1, (SELECT count(*) FROM menu_items) AS actual;

-- Cleanup
RESET ROLE;
DELETE FROM menu_items WHERE name IN ('Burger A', 'Burger B');
DELETE FROM restaurants WHERE name IN ('Tenant A Test', 'Tenant B Test');
```

Note: The exact test SQL above uses placeholder approach because `SET request.jwt.claims` is the canonical Supabase pattern for impersonating a JWT in psql/SQL Editor. Document in the README that this script requires manual UUID substitution in two `<REST_A_UUID>` placeholders during run.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser/anon JWT -> Supabase DB | Untrusted JWT crosses; RLS enforces restaurant_id claim — bypass means tenant data leak. |
| Supabase Auth (token mint) -> custom_access_token_hook -> JWT | Hook reads from `restaurants` and writes claim; bug here = wrong tenant assigned. |
| Postgres `authenticated` role -> tenant rows | Without RLS, an authenticated user can read all tables; this is the highest-impact threat in the project. |
