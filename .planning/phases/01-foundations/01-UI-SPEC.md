# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundations.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> Phase 1 UI scope is intentionally minimal: three pages only.
> Login · Signup · Dashboard placeholder (empty state).
> All product UI (menu editor, KDS, onboarding wizard) lives in Phases 2–4.

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Form helper text, descriptive copy |
| Label | 14px | 500 (medium) | 1.4 | Input labels, nav items |
| Heading | 20px | 600 (semibold) | 1.2 | Page headings ("Crear cuenta", "Iniciar sesión") |
| Display | 28px | 700 (bold) | 1.1 | App name / brand in auth pages only |

Font family: `Inter, ui-sans-serif, system-ui, -apple-system, sans-serif`

**Constraint:** Only 2 weights in use during implementation — regular (400) and semibold (600).
The medium (500) label weight maps to semibold in practice if the font does not render
a distinct 500 step. Bold (700) is display-only, used once per auth page.

Source: defaults based on standard SaaS B2B dashboard patterns.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f0f0f` (near-black) | Page background, outermost surface |
| Secondary (30%) | `#1a1a1a` (dark card) | Auth card background, form container |
| Accent (10%) | `#f97316` (orange-500) | Primary CTA button only |
| Destructive | `#ef4444` (red-500) | Error alerts, form validation errors |

**Rationale for dark theme in Phase 1:**
Phase 4 KDS (kitchen display) is dark-mode by default (KDS-08). Starting the design
system in dark mode from Phase 1 means the executor does not migrate colors later.
The restaurant operator context (late-night operations, glare reduction) also favors dark.

**Accent reserved for:**
- Primary submit button on Signup page ("Crear cuenta")
- Primary submit button on Login page ("Ingresar")
- No other elements in Phase 1 use the orange accent.

**Secondary semantic colors:**
- `#22c55e` (green-500): reserved for Phase 2+ success states (not used in Phase 1)
- `#f97316` stays off of links, borders, hover states — buttons only.

**Tailwind CSS custom property mapping** (set in `apps/frontend/src/index.css`):
```css
:root {
  --background: 0 0% 6%;           /* #0f0f0f */
  --card: 0 0% 10%;                /* #1a1a1a */
  --primary: 24 95% 53%;           /* #f97316 orange-500 */
  --destructive: 0 84% 60%;        /* #ef4444 red-500 */
  --foreground: 0 0% 95%;          /* #f2f2f2 near-white text */
  --muted-foreground: 0 0% 60%;    /* #999 subtle labels */
  --border: 0 0% 18%;              /* #2e2e2e subtle borders */
  --ring: 24 95% 53%;              /* focus ring = accent */
}
```

Source: defaults chosen to match Phase 4 KDS dark requirement (KDS-08, ROADMAP.md).

## Copywriting Contract

Language: **Spanish Rioplatense (Argentine informal "vos")**. No "tú". No "usted" in UI
copy (product is casual B2B for restaurant owners, not corporate).

| Element | Copy |
|---------|------|
| Primary CTA — Login | "Ingresar" |
| Primary CTA — Signup | "Crear cuenta" |
| Secondary action — Login page | "¿No tenés cuenta? Registrate" |
| Secondary action — Signup page | "¿Ya tenés cuenta? Iniciá sesión" |
| Forgot password link | "Olvidé mi contraseña" |
| Post-signup instruction | "Revisá tu email para confirmar tu cuenta" |
| Post-reset instruction | "Te enviamos un link para restablecer tu contraseña" |
| Email verification pending | "Confirmá tu email para continuar" |
| Dashboard empty state heading | "Bienvenido a Agente Restaurante" |
| Dashboard empty state body | "Tu workspace está listo. Configurá tu restaurante para empezar a tomar pedidos." |
| Dashboard empty state CTA | "Configurar restaurante" (leads to Phase 2 onboarding — placeholder link in Phase 1) |
| Error — invalid credentials | "Email o contraseña incorrectos. Intentá de nuevo." |
| Error — email already registered | "Ya existe una cuenta con ese email. Iniciá sesión." |
| Error — weak password | "La contraseña debe tener al menos 8 caracteres." |
| Error — network | "No pudimos conectarnos. Revisá tu conexión e intentá de nuevo." |
| Error — generic fallback | "Algo salió mal. Intentá de nuevo o contactanos si el problema persiste." |

**Destructive actions in Phase 1:** none. Logout is present (AUTH-07) but not destructive —
no confirmation dialog required. Button label: "Cerrar sesión".

**Tone:** Direct, warm, concise. No exclamation marks in errors. No passive voice.
Example of what NOT to write: "Se ha producido un error en el proceso de autenticación."
Example of what TO write: "Algo salió mal. Intentá de nuevo."

Source: CONTEXT.md (Spanish rioplatense context), product ICP (Argentine restaurant owners).

## Page Layout Contracts

### Login page (`/login`)

- Full-viewport centering: `min-h-screen flex items-center justify-center`
- Background color: dominant (`--background`)
- Auth card width: `w-full max-w-sm` (384px)
- Card padding: 32px (xl token)
- Vertical stack inside card: Display (app name) → Heading → email input → password input → CTA button → secondary link
- App name display: "Agente Restaurante" at 28px bold, centered, `--foreground`
- Page heading: "Iniciá sesión" at 20px semibold, centered
- No logo/icon in Phase 1 (visual identity deferred)

### Signup page (`/signup`)

- Same layout as Login
- Heading: "Crear tu cuenta"
- Fields: email → password → confirm password (3 inputs)
- After submit success: show inline success message (not a new page) — "Revisá tu email para confirmar tu cuenta." CTA button replaced with the message.

### Dashboard placeholder (`/dashboard`)

- Full page layout with minimal nav bar (top): height 56px, background `--card`
- Nav bar content: app name left-aligned (14px semibold) + "Cerrar sesión" button right-aligned (ghost variant)
- Main content: centered empty state block — heading + body + placeholder CTA
- Empty state block: `max-w-md`, centered, `gap-lg` between elements
- No sidebar in Phase 1 (sidebar is Phase 2+ concern)

## Interaction Contracts

### Form validation

- Validate on submit (not on blur) — reduces noise during typing.
- Show `<Alert variant="destructive">` above the CTA button for auth errors returned by Supabase.
- Show inline error below each input (red-500 text, 12px, `--destructive` color) for client-side validation (empty field, password mismatch, weak password).
- Error messages must use copywriting contract copy exactly.

### Auth redirect logic

| Condition | Redirect |
|-----------|----------|
| Logged in, `restaurant_id` in JWT | `/dashboard` |
| Logged in, `restaurant_id === null` in JWT | `/dashboard` (empty state — Phase 2 onboarding handles the next step) |
| Not logged in, visits `/dashboard` | `/login` |
| Successful logout | `/login` |

Source: RESEARCH.md Pattern 1 (Pitfall 2 — JWT without restaurant_id on first login).

### Accessibility baseline

- All inputs have associated `<Label>` with `htmlFor` — required.
- Focus ring uses `--ring` (orange-500) — visible against dark background.
- Error alerts have `role="alert"` — screen reader announcement on error.
- Minimum contrast ratio: 4.5:1 for body text (`#f2f2f2` on `#1a1a1a` = 12.6:1, passes AAA).
- Tab order: email → password → CTA → secondary link.
