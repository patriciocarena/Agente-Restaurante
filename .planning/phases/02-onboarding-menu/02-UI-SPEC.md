## Spacing Scale

Inherited from Phase 1 — no changes.

| Token | Value | Usage in Phase 2 |
|-------|-------|------------------|
| xs | 4px | Icon gaps inside item rows, option-item row inner padding |
| sm | 8px | Compact stacks (label + input, switch + label) |
| md | 16px | Default form field gap, card inner padding, modal section padding |
| lg | 24px | Step content gaps, Settings tab content padding, MenuEditor column padding |
| xl | 32px | Wizard step block vertical rhythm, modal outer padding |
| 2xl | 48px | Empty state vertical centering offset |
| 3xl | 64px | Page-level top/bottom centering for Onboarding |

**Exceptions for Phase 2:**

- **Touch targets in MenuEditor item rows:** minimum 44×44px hit area for the availability `Switch` and the "Editar" button, since `/menu` may be used from a tablet by the restaurant owner mid-shift. Implementation: wrap interactive elements in a `min-h-11` row (44px) and apply `px-4` (16px) horizontal padding. Switch component itself is smaller; the row hit area satisfies the target.
- **Wizard stepper bar:** fixed 56px height (matches Phase 1 dashboard nav) — not on the 4px scale strictly, but consistent with the existing app shell. Documented as a deliberate exception.

## Typography

Inherited from Phase 1 (4 sizes, exactly 2 weights). Phase 2 reuses the same roles:

| Role | Size | Weight | Line Height | Usage in Phase 2 |
|------|------|--------|-------------|------------------|
| Body | 14px | 400 (regular) | 1.5 | Helper text, descriptions, option-item labels, item prices in list view |
| Label | 14px | 600 (semibold) | 1.4 | Field labels, category names in sidebar, item names in list, tab labels in Settings |
| Heading | 20px | 600 (semibold) | 1.2 | Step heading ("Datos del restaurante", "Horario", etc.), modal title, MenuEditor section heading |
| Display | 28px | 600 (semibold) | 1.1 | Empty state heading ("Tu menú está vacío") only |

**Constraint preserved:** Exactly 2 weights — regular (400) + semibold (600). Do not introduce medium (500) or bold (700).

**Numeric/price formatting:** Prices render as `$1.500` (AR convention: dot as thousands separator, no currency code). Use `toLocaleString('es-AR')`. Price text uses Body role (14px regular) in list view; uses Label role (14px semibold) inside the item edit modal as a focused field value.

**Spanish characters:** Inter has full coverage for ñ, á, é, í, ó, ú, ü. No font swap needed.

## Color

Inherited from Phase 1 — `index.css` variables remain authoritative.

| Role | Value | Usage in Phase 2 |
|------|-------|------------------|
| Dominant (60%) | `#0f0f0f` (`--background`) | Page background everywhere |
| Secondary (30%) | `#1a1a1a` (`--card`) | Wizard card, MenuEditor sidebar, item rows, modal background, Settings tabs panel |
| Accent (10%) | `#f97316` (`--primary`) | See "Accent reserved for" below |
| Destructive | `#ef4444` (`--destructive`) | Validation errors, "Borrar todo" button in delete-cascade modal, Twilio failure state |
| Foreground | `#f2f2f2` (`--foreground`) | Primary text |
| Muted foreground | `#999` (`--muted-foreground`) | Helper text, "Cerrado" day label, sold-out item label, placeholders |
| Border | `#2e2e2e` (`--border`) | Card borders, input borders, sidebar/main divider |

**Accent reserved for** (explicit allowlist — no other element may use orange):

1. Primary CTA button in each wizard step ("Siguiente", "Terminar")
2. Primary CTA button "Guardar cambios" in Settings tabs
3. Primary CTA button "Guardar" inside item edit modal
4. Primary CTA button "Cargar template hamburguesera" in empty state
5. Focus ring (`--ring` system token) on all focusable elements
6. The active step indicator pill in the wizard stepper bar (current step only — completed steps use muted foreground)
7. The active tab indicator in Settings tabs (bottom border 2px on active tab)
8. The thumb of the availability `Switch` when ON (Radix Switch default — already handled by shadcn primary token)

**Accent NOT allowed on:**

- Item names, prices, descriptions
- Category names in sidebar (active category uses `--card` background tint, not orange text)
- "Reintentar" button on Twilio failure (use default ghost variant — orange is reserved for forward-progress CTAs only)
- Hover states (use 5% white overlay via Tailwind `hover:bg-white/5`, not orange)
- Borders, links, icons (except inside primary CTA buttons)

**Secondary semantic colors:**

- `#22c55e` (green-500) — reserved for future success toasts, not used in Phase 2.
- `#ef4444` (`--destructive`) — used ONLY for: inline form errors, the "Borrar todo" button in the delete-cascade confirmation modal (D-15), the Twilio failure pane background tint (`bg-destructive/10` border), and the validation error alert at the top of a wizard step when submit fails.

**Switch (availability toggle) color treatment:**

- ON state: thumb on `--primary` track (orange). Visually says "available".
- OFF state: thumb on `--border` track. Visually says "sold out".
- Item row when OFF: row opacity `0.6` + the item name strikethrough is **NOT** used (avoid visual noise — opacity alone is the signal). A muted "Sin stock" pill (Body 12px, `--muted-foreground`) appears next to the price.

## Interaction Contracts

### Wizard navigation

- "Siguiente" calls `methods.trigger(stepFields)` (RHF). On valid: UPSERT to backend (PATCH `/api/restaurants/me` for steps 1-4 partial), set `onboarding_step` = current+1, advance the step state.
- On invalid: errors render inline below each field; the inline error uses Body 12px in `--destructive`. NO toast — inline is enough.
- "Atrás" never validates — it just decrements step state. Form values are preserved (single `useForm` instance per Pattern 1).
- Stepper bar segments are NOT clickable for forward navigation (must complete in order). They ARE clickable for backward navigation to revisit a completed step.

### Slug auto-derivation

- On Step 1 name input blur or first slug field focus (whichever first), auto-fill slug using `slugify(name, { lower:true, strict:true, locale:'es' })`.
- If the user manually edits slug after auto-derivation, freeze auto-derivation (don't overwrite on subsequent name edits).
- On submit, backend checks uniqueness; if taken, returns 409 with `{ error: 'slug_taken' }` → frontend renders inline "Slug taken" error on the slug field.

### Form validation timing

- Validate on submit only (Phase 1 carry-forward), NOT on blur. Reduces noise during typing.
- Exception: the slug field validates its regex inline on change (instant feedback because the user can introduce invalid characters easily — `Ñ`, spaces, etc.). Other fields wait for submit.

### Availability toggle (MENU-04)

- Switch click → optimistic UI flip + PATCH `/api/menu-items/:id/availability`.
- On success (Realtime echo): no UI change (already optimistic).
- On failure (network/backend error): revert the Switch + show toast "El cambio no se guardó. Probá de nuevo."
- Cross-tab sync: Supabase Realtime subscription updates other open tabs/devices within <2s (NFR MENU-04). When a remote update arrives, the row briefly highlights with a `bg-primary/5` flash for 400ms then fades — a subtle "synced" signal.

### Item edit modal save

- Save triggers a single backend transaction (POST or PATCH item + diff of option_groups/option_items). Frontend shows the "Guardar" button in loading state (Loader2 icon + disabled) during the call.
- On success: close modal + show toast "Item guardado".
- On failure: keep modal open + render error alert above the footer ("No pudimos guardar. Revisá tu conexión e intentá de nuevo.").

### Delete category with items (D-15)

- Trash icon on category row → fetch item count → open Dialog with D-15 copy showing the count.
- Confirm "Borrar todo" → DELETE `/api/menu-categories/:id` (DB CASCADE handles items) → close dialog → remove category from sidebar + clear main column → show toast "Categoría borrada".

### Mid-shift sold-out flow (the core value moment)

- User opens `/menu` on tablet → tap Switch on an item → row visually marks unavailable in <100ms (optimistic) → backend confirms in <500ms typical → other devices see the change in <2s via Realtime.
- This is the most-used interaction in the post-MVP lifecycle of the product. The Switch must be reachable in 1 tap from `/menu`. No drill-down to item modal required.

## Accessibility Baseline

- All inputs have associated `<Label>` with `htmlFor` (Phase 1 standard).
- Focus ring `--ring` (orange-500) visible on all interactive elements against the dark background.
- Wizard stepper: `role="progressbar"` with `aria-valuenow` (current step 1-indexed), `aria-valuemax="4"`, `aria-valuetext="Paso 2 de 4: Horario"`.
- Each step heading is an `<h1>` to ensure screen readers announce the step transition. The page-level title is suppressed (visually hidden) on wizard pages.
- Dialog: `role="dialog"`, `aria-labelledby` pointing to the modal title, `aria-modal="true"`. ESC closes (except item edit modal — see Interaction Contracts).
- Switch: native `role="switch"` from Radix. `aria-label="Disponibilidad de {item.name}"` so screen reader announces context.
- Trash buttons: `aria-label="Borrar categoría {name}"` / `aria-label="Borrar item {name}"` since the visual is icon-only.
- Drag handle: `aria-label="Reordenar categoría"`. Keyboard alternative: arrow keys when the row is focused (defer to Phase 3 if not trivial — pilot uses mouse/touch).
- Color contrast: all body text 4.5:1+ on its background. Verified Phase 1: `#f2f2f2` on `#1a1a1a` = 12.6:1, `--muted-foreground #999` on `#0f0f0f` = 6.4:1.
- Touch targets: min 44×44px for all primary interactions on MenuEditor (see Spacing exceptions).
- Tab order in wizard: skip the back button when on step 1; ensure Tab cycles through visible fields then to "Siguiente".

## Realtime Visual Behavior (MENU-04)

When a `menu_items` UPDATE arrives via Supabase Realtime channel `menu-{restaurantId}` and the item belongs to a category currently visible in the main column:

1. The corresponding row receives a one-time CSS class `animate-flash-primary` for 400ms.
2. The Switch state updates to the new value if it differs from current visual state.
3. The price / name / availability updates instantly without any user action.

The `animate-flash-primary` animation: background fades from `bg-primary/5` to transparent over 400ms via `@keyframes` in `index.css`. Keep it subtle — this is a confirmation, not an alert.
