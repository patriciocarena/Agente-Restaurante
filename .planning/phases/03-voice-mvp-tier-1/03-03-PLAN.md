<objective>
Build the webhook — the single most important file in the phase. It receives Vapi's confirm_order tool-calls and end-of-call-report events, secures by X-Vapi-Secret, routes by assistantId to a tenant, enforces business hours server-side, validates items against the live menu, recalculates all prices server-side (never trusts the LLM), assigns a per-tenant order number atomically, and persists the order idempotently. The transcript (only available at end-of-call) is written back onto the order (CALL-09).

Purpose: This is the core-value path — "el pedido llega a la cocina sin que nadie atienda." Every CALL-0x and OBS-01 requirement lands here.
Output: `routes/vapi-webhook.ts` mounted in `index.ts`; vapi-webhook.test.ts goes GREEN.

  order_items: order_id, restaurant_id, menu_item_id, name, quantity int, unit_price int, modifiers jsonb, note.
  menu_items: id, restaurant_id, category_id, name, base_price int, available boolean.
  option_groups (per menu_item) -> option_items (name, price_delta int) — modifier price source.
  restaurant_counters: increment_order_counter(p_restaurant_id uuid) RETURNS int  (RPC from Plan 01).
  restaurant_hours: day_of_week (0=Dom..6=Sab), open_time time, close_time time, is_closed boolean.
  call_logs: restaurant_id, call_id UNIQUE, order_id, duration_seconds, cost_usd numeric, transcript, ended_reason, started_at, ended_at.

Vapi webhook payload (verified SDK types):
  tool-calls: message.type==='tool-calls'; message.toolCallList[0].id = toolCallId;
    message.toolCallList[0].function.arguments = { items, fulfillment_type, customer_name, delivery_address? };
    message.call.id = call_id; message.call.assistantId = routing key; message.customer?.number = phone (PII, NULL on web calls).
    RESPONSE shape: { results: [{ toolCallId, result: string }] }
  end-of-call-report: message.type==='end-of-call-report'; message.cost (USD);
    message.artifact?.transcript; message.call.startedAt/endedAt (ISO); message.endedReason; message.call.id.
